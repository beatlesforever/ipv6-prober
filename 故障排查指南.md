| ---------------- | -------------------------------------- |

> 探测方向：服务器 → Windows 本机

***

## 排查逻辑

对每种探测类型，按链条逐环节排查：

```
[1] 报文构造正确？ → [2] 服务器发出了？ → [3] Windows 收到了？ → [4] Windows 回复了？ → [5] 服务器收到回复？
```

**任一环节中断，就能定位问题根因。**

### 2-5 判断

| 环节                                      |  状态 |
| --------------------------------------- | :-: |
| 报文构造正确                                  |  ✅  |
| 服务器 tcpdump 抓到 N 个分片 (5个)               |  ✅  |
| Windows pktmon 收到 N 个分片 (Rx, 5个)        |  ✅  |
| Windows pktmon 回复分片 Echo Reply (Tx, 2个) |  ✅  |
| 服务器收到 Echo Reply                        |  ✅  |

**结论**：IPv6 分片完全穿透，路径没有任何分片过滤。RTT=304ms (normal 266ms, 分片重组有少量额外开销), hlim=45 (与 normal 一致)。

> **出站 5 片 vs 回复 2 片**：不是对等关系。服务器按 `fragSize=512` 把大包切成 5 片发出，Windows 收到后重组为完整 ICMPv6 Echo Request，然后回复一个同样大的 Echo Reply。Windows 按自己的 MTU (\~1500 字节，每片最多装 \~1448 字节 IPv6 payload) 把回复切成 2 片发回。两边分片策略不同，总 payload 相同 (\~2037 字节)。

## 测试 3：spoofed-src（伪造源地址）

**目的**：测试 BCP38 入口过滤。

> 本测试分两组：实验组（伪造源地址 `2001:db8:a001:0:0:0:0:1`）和对照组（用服务器真实地址走同一代码路径）。
>
> **实验组**：源地址不属于服务器，如果 Windows 能收到，说明路径没有 BCP38 过滤。服务器不期望收到回复（回包被路由到伪造地址去了）。
> **对照组**：源地址就是服务器自己，代码路径和实验组完全一样，用来排除代码本身的问题。服务器应收到回复。
